## 1. Design context & assumptions

- **Audience:** Content editors, marketers, Sitecore solution engineers, and QA reviewers working inside the SitecoreAI Pages editor. They are knowledge workers who live in editorial/content tooling all day; a substantial portion have a design eye and notice typographic care. The panel is used briefly but repeatedly — several times per day, per user.
- **Goals and success criteria:**
  - Make the captured page feel like a **preserved artifact** rather than a utility-dump image, so the tool reads as respectful of the work, not a commodity screen-grabber.
  - Keep the four PRD states (idle, capturing, ready, error) legible and recoverable inside a 320–400 px wide column with no horizontal scroll.
  - Hit PRD M1–M4 (≤ 10 s capture, ≤ 3 clicks end-to-end, no editor chrome in output, actionable error states) and NFR-A-01 (WCAG 2.1 AA, keyboard, ARIA live regions).
- **Constraints (brand, a11y, stack):**
  - **Surface:** Sitecore Pages Page Builder Context Panel — a narrow vertical iframe column, 320–400 px wide, vertically scrollable, embedded in a light-themed editor chrome.
  - **Tech:** Next.js 15+ App Router, React 19, TypeScript strict, Tailwind CSS only (ADR-0004). Radix UI / shadcn/ui primitives permitted.
  - **State:** client is stateless beyond React state; no persisted history; no secrets client-side (ADR-0002, ADR-0006). This direction therefore cannot fake a "gallery of past captures" — the archival feel must come from presentation, not persistence.
  - **Accessibility:** WCAG 2.1 AA; keyboard-first; visible focus rings; ARIA live region for state transitions; respects `prefers-reduced-motion`.
  - **Brand:** No established PageShot brand. This variant commits to an editorial/archival identity that the product can adopt if chosen.
- **Reference / inspiration:**
  - **Framing & passepartout tradition** — museum-grade matted prints: a generous neutral border that isolates the image and signals "this is the work." See [WhiteWall — Picture Design with Passepartouts](https://www.whitewall.com/us/magazine/inspiration/passepartout), [HALBE — How to find the perfect passepartout](https://www.halbe-rahmen.de/en/know-how/passepartout/).
  - **Serif revival in editorial UI** — [Creative Boom — 50 fonts for 2026](https://www.creativeboom.com/resources/top-50-fonts-in-2026/), [Fontfabric — 10 design trends 2026](https://www.fontfabric.com/blog/10-design-trends-shaping-the-visual-typographic-landscape-in-2026/), [DesignMonks — Typography Trends 2026](https://www.designmonks.co/blog/typography-trends-2026). Serif-for-interface is a live 2026 pattern, not nostalgia cosplay.
  - **Editorial captioning & metadata as first-class** — [Montilla Design — Static Captions in InDesign](https://www.montilladesign.com/post/learn-how-to-generate-static-captions-in-adobe-indesign-layouts), [Design4Users — Elegant Minimalist Editorial Designs](https://design4users.com/less-is-more-elegant-and-minimalist-editorial-designs/).
  - **Photographer-archive minimalism** — [Readymag portfolios](https://readymag.com/portfolio), [Readymag blog — Made with Readymag](https://blog.readymag.com/made-with-readymag-templates/): one typeface, strict grid, order-reduction-clarity.
  - **Quieter weblog voice** — [kottke.org on typography](https://kottke.org/tag/typography) as a tonal touchstone for a tool that respects its content.
- **Answers to clarifying questions** *(designer's-call resolutions):*
  - *Color palette?* Warm neutral paper tones — cream/bone card on a very pale stone background. Ink is near-black, never pure black. Dividers are hairline stone. Single restrained accent: an oxblood/iron-red reserved for error and active-capture states. Deliberately not corporate-blue.
  - *Dark mode?* **No** in v1. Dark mode fights the passepartout metaphor (a lit, matted frame on a bright wall) and the Sitecore Pages editor is light-themed. Revisit as a follow-on.
  - *Typography?* Serif-forward. Display/body in a readable text-serif (Source Serif 4) paired with a grotesk for metadata captions, buttons, and state labels (Inter). Serif at body size is the 2026 editorial pattern, but small-size legibility is a known risk at 320 px — mitigated with minimum 14 px body and 13 px metadata (see § 5).
  - *Motion?* Minimal and slow. Capture spinner is a rotating hairline circle, not a jittery chase. State transitions use 200 ms ease-out opacity fades. All motion collapses to instant under `prefers-reduced-motion`.
  - *Iconography?* Hairline 1.25 px stroke icons (Lucide, overridden), same weight as the frame rules. No filled glyphs. Capture button uses a small aperture/diaphragm mark rather than a camera icon.
  - *Tone of copy?* Understated, complete sentences, no exclamation. Error copy is directive but polite ("Save the page, then try again.").
  - *"Artifact-ness" with no persistence?* The current capture is presented as **Plate I** of an implicit archive — a single exhibit card with an index number, not a gallery. No promise of history; just dignity for this one image.
- **Assumptions made:**
  - A1 — The Sitecore Pages editor chrome around the iframe is light-neutral; our off-white panel will not clash.
  - A2 — Screenshots returned by the Agent API are rendered web pages of varying aspect ratios; the frame must accommodate portrait, landscape, and very tall pages.
  - A3 — The `pageName` and `siteName` from `pages.context` are display-safe strings of reasonable length; we will still clamp to 2 lines.
  - A4 — Serif text at 14–15 px on a high-DPI editor display is readable. If user testing at 320 px shows otherwise, the swap to Inter-only is a single token change (see § 6 open question).
  - A5 — `prefers-reduced-motion` is respected at CSS level without JS feature detection.
  - A6 — `navigator.clipboard.write` with `ClipboardItem` is available on modern Chromium/WebKit; Firefox fallback hides the Copy button or shows the documented denial message per AC-2.4.

## 2. Design direction

- **Visual direction: "Passepartout" — the framed-archive panel.**

  The panel is staged like a single-exhibit card in a museum digital collection. Generous cream "paper" fills the viewport; a hairline stone border inset by uniform margin acts as the passepartout. Inside that mat sits the capture plate (the preview or its placeholder), with an index mark above ("Plate I") and a typeset caption beneath — site name set in serif italic, page name in serif roman, and a small-caps metadata line with timestamp and dimensions. Buttons sit below the caption as quiet directives, not calls-to-action. The capture button is the only element that blooms — its border thickens on focus and its label sets in serif small caps to signal gravity.

  The direction **commits to these choices**: warm neutrals, single restrained accent, serif-forward typography, hairline rules instead of shadows or gradients, metadata treated as caption craft, and composition over decoration. The tool respects the content it captures — that's the whole thesis.

  - **Named tokens (intent):**
    - `bg-stone-50` — wall / outer background.
    - `bg-[#FAF7F0]` (card "bone") — the passepartout paper.
    - `text-stone-900` — ink for body and display.
    - `text-stone-500` — muted metadata, captions' secondary line.
    - `border-stone-300` — hairline frame rule.
    - `accent-[#7A1E1E]` (oxblood) — error and in-flight "live" indicator only.
    - `font-serif` → Source Serif 4 (fallback: Georgia, "Times New Roman", serif).
    - `font-sans` → Inter (fallback: system-ui) — used only for captions' small-caps line, button labels, and state messages.
  - **Motion intent:** 200 ms opacity fades on state change; 1.2 s linear spin on the capturing indicator; `motion-reduce:transition-none motion-reduce:animate-none` everywhere.

- **Layout and structure** (from top to bottom, within the panel iframe):

  1. **Masthead** — fixed-at-top cream band with hairline bottom rule. Contains the product mark set in serif small caps: `P A G E S H O T` with a 1 px hairline underline flourish below. No logo mark — typography is the mark. Height ~44 px.
  2. **Exhibit card** — the mat. Scrolls with the page. Outer padding ~20 px left/right of card. Card has hairline `border-stone-300`, cream fill, ~16 px inner padding, no shadow.
     - **Plate header** — tiny serif italic `Plate I` (Roman-numeral always "I" in v1 since no history), right-aligned, above the preview. Communicates "this is the current exhibit" without promising a gallery.
     - **Preview frame** — the actual image sits inside a second hairline rule with 8 px inner mat. Fixed aspect-ratio container letterboxes the image into the available width, preserving the returned image's aspect ratio via `object-contain`. A minimum height (~180 px) guarantees presence in idle/error states.
     - **Caption block** — directly under the preview, left-aligned, three lines:
       - Line 1: `<SiteName>` set in serif italic, 13 px small-caps letter-spacing.
       - Line 2: `<PageName>` set in serif roman, 16 px, tight leading.
       - Line 3: small-caps sans line: `CAPTURED 22 APR 2026 · 09:14 · 1440 × 3200` (dimensions filled when known; omitted otherwise).
     - **Hint rule** — a hairline followed by the static hint from FR-12 in serif 13 px italic: *"The plate shows the last saved version of this page."* (Rewritten from PRD's "Screenshot shows…" to match the metaphor; same semantic content.)
  3. **Action strip** — outside the card. Three quiet buttons stacked vertically at 320 px, paired 2-up below 360 px, left-aligned labels:
     - Primary: `Capture` — serif small-caps label inside a bone-card button with a thicker (1.5 px) stone-700 border.
     - Secondary: `Copy image` — sans label, hairline border, transparent fill.
     - Secondary: `Download PNG` — sans label, hairline border, transparent fill.
  4. **Foot rule** — hairline, then a muted sans line: `An authoring-time capture tool. No images are stored.` (Reinforces ADR-0006's privacy posture in a way that fits the editorial voice.)

- **Key components and patterns:**
  - `<Masthead>` — typographic mark band.
  - `<ExhibitCard>` — the passepartout card with slots for `plateLabel`, `preview`, `caption`, `hint`.
  - `<PreviewFrame>` — aspect-preserving image mat with four visual states (idle/capturing/ready/error).
  - `<Caption>` — three-line editorial caption with site/page/meta props.
  - `<ActionButton>` — two variants: `primary` (bone fill, thick border, serif small caps) and `quiet` (transparent, hairline border, sans label).
  - `<ElapsedTimer>` — serif-italic running-seconds line that appears after 5 s (FR-10).
  - `<InlineMessage>` — one-line notice under a button or inside the frame for errors, copy-confirmation, and offline ("Copied to clipboard." / "This page has not been saved yet. Save it first, then try again.").

## 3. Key screens and flows

This is one route (`app/panel/page.tsx`) with four visual states. Each state is a full render of the panel.

- **Screens (states):**
  1. **S-IDLE — "The empty mat."** Exhibit card present, preview frame shows a placeholder mat with a centered hairline aperture glyph and the line *"Click Capture to take a plate of this page."* in serif italic. Capture button enabled; Copy/Download disabled (rendered but at 40 % opacity, `aria-disabled="true"`).
  2. **S-CAPTURING — "Exposing the plate."** Preview frame dims 8 % and shows a centered hairline spinning circle (aperture motif) plus the text `Exposing the plate…` in serif italic. Capture button disabled. After 5 s, an `<ElapsedTimer>` appears under the caption block: `Still exposing · 6 s · 7 s …`. ARIA live region (`aria-live="polite"`) announces "Capture in progress" at start.
  3. **S-READY — "Plate I mounted."** Image fills the preview frame via `object-contain`. Caption lines populate with real site/page/timestamp/dimensions. Capture, Copy, Download all enabled. A transient `<InlineMessage>` reading *"Copied to clipboard."* appears under the Copy button for 2 s after a successful copy, then fades (AC-2.3). Successive captures replace Plate I in place (FR-11).
  4. **S-ERROR — "Plate unavailable."** Preview frame shows a plain cream mat with a small oxblood hairline corner mark (top-right) and the error message inside the frame in serif 15 px roman. Each PRD error code maps to a message (see § 4). A `Retry` quiet button appears as a fourth action at the top of the action strip, above `Capture`. Copy/Download disabled. Static hint still visible (US-4).

- **Critical user flows:**
  1. **Happy path (US-1 → US-2 or US-3):** Panel mounts → S-IDLE → user presses `Capture` (mouse or Enter when focused) → S-CAPTURING → response arrives → S-READY → user presses `Copy image` → clipboard write → InlineMessage "Copied" for 2 s. Alternative: user presses `Download PNG` → browser download with templated filename (FR-09).
  2. **Slow capture (FR-10):** S-CAPTURING → at 5 s, `<ElapsedTimer>` joins the caption block and increments every second → on response, S-READY (timer removed). ARIA live announces "Still working" once, not every second.
  3. **Error & retry (US-5):** S-CAPTURING → error envelope → S-ERROR with code-specific message → user presses `Retry` → S-CAPTURING. If the error is `auth`, Retry is still offered (AC-5.5) because retry might succeed after token refresh.
  4. **Unsaved page (US-4):** Static hint visible in all states. On a `not_found` / 404, S-ERROR message reads *"This page has not been saved yet. Save it first, then try again."* (AC-4.3).
  5. **Keyboard-only:** Tab order is `Capture → Copy → Download` in S-READY; `Capture` → `Retry` in S-ERROR. Enter/Space activates. Esc in S-READY returns to S-IDLE (optional per PRD).

- **Responsiveness (hard floor 320 px, hard ceiling 400 px, vertically scrollable):**
  - **320 px (floor):** Single-column action strip (Capture, Copy, Download stacked). Card inner padding reduces to 12 px. Caption page-name wraps to 2 lines if needed; line-clamp at 3. Masthead mark kerning reduces from 0.3 em to 0.22 em. Body serif stays 15 px; metadata sans stays 13 px — minimum per § 5 legibility guard.
  - **360 px (target):** Card inner padding 16 px. Action strip: Capture on its own row; Copy and Download share a row. Caption metadata line fits on one line for typical site/page names.
  - **400 px (ceiling):** Card inner padding 20 px. All three buttons may share one row if labels fit (`Capture` / `Copy` / `Download` — note: shortened labels in this breakpoint only, still accessible-named). Otherwise fall back to 360 px pairing.
  - **Vertical:** No fixed-height constraint. Card grows with content. Masthead is sticky at the top of the iframe (`sticky top-0 bg-[#FAF7F0]/95 backdrop-blur-sm`) so the mark stays visible on scroll. Foot rule scrolls with the card.
  - **Image aspect fidelity:** Returned screenshots of tall pages are letterboxed into the frame via `object-contain` with max-height `min(70vh, 560px)`. A hairline is never cropped off the image.

### 4.3 Accessibility notes

- **Focus pattern:** all interactive elements use `focus-visible:ring-2 ring-stone-900 ring-offset-2 ring-offset-[#FAF7F0]`. Focus rings are ink-black on bone, ~4.5:1 contrast against the card — clearly visible. Never rely on border-color change alone.
- **ARIA live:** an `AriaLive` (`role="status"`, `aria-live="polite"`) announces state transitions:
  - S-IDLE → S-CAPTURING: "Capturing screenshot."
  - 5 s elapsed in S-CAPTURING: "Still capturing." (announced **once**, not per second — prevents screen-reader spam).
  - S-CAPTURING → S-READY: "Screenshot ready." plus dimensions if present.
  - Any error: the error message text.
  - Copy success: "Copied to clipboard."
- **Semantics:** the capture action is `<button type="button">`. The preview `<img>` has a descriptive `alt`. The plate label `Plate I` is decorative (`aria-hidden="true"`).
- **Color contrast:**
  - Ink `#1C1917` on bone `#FAF7F0` → 14.8 : 1 (AAA body).
  - Muted `#57534E` (stone-600) on bone → 7.2 : 1 (AAA body, AA small text).
  - Secondary `#78716C` (stone-500) on bone → 4.6 : 1 (AA large text / metadata ≥ 18 px-equivalent or 14 px bold; our 11 px small-caps is at the edge — see § 5 mitigation).
  - Oxblood `#7A1E1E` error on bone → 8.4 : 1 (AAA body). Also not used as the *sole* signal — an icon and the text "Retry" accompany it.
- **Keyboard:** visible focus ring on every control. Tab order: Masthead-skip → `Capture` → (`Copy` → `Download`) in S-READY; `Retry` → `Capture` in S-ERROR. Enter/Space activate. Esc in S-READY clears to S-IDLE (optional, nice-to-have per PRD § 11). No keyboard traps.
- **Reduced motion:** All `animate-spin`, `transition-opacity`, and skeleton-fade use `motion-reduce:animate-none motion-reduce:transition-none`. The capturing state then presents a static aperture glyph with the label "Exposing…" — still informative, no motion.

## 5. Responsiveness and accessibility

- **Breakpoints** (the panel only — the Tailwind `sm` default of 640 px is never reached inside the iframe):
  - **320 px floor:** single-column buttons; card padding 12 px; masthead letter-spacing 0.22 em; action strip spans full width minus 16 px outer margin.
  - **360 px target:** card padding 16 px; Copy + Download share a row.
  - **400 px ceiling:** card padding 20 px; all three buttons can share a row if labels fit.
  - Internal breakpoints implemented via container queries (`@container (min-width: 360px)`) so the panel responds to its iframe width, not the outer viewport.

- **WCAG 2.1 AA requirements:**
  - **1.3.1 Info and Relationships:** semantic `<header>`, `<main>`, `<figure>`/`<figcaption>` for preview + caption, `<button>` for actions.
  - **1.4.3 Contrast:** all body/UI text ≥ 4.5 : 1 against bone; the 11 px metadata line is treated as decorative supplementary info — the same timestamp/dimensions are also announced through the ARIA live message on S-READY so they are not the *sole* channel for that data.
  - **1.4.11 Non-text Contrast:** borders on all buttons are ≥ 3 : 1 against the card fill (stone-400 = 3.1 : 1; stone-700 primary border = 8.6 : 1).
  - **1.4.12 Text Spacing:** no fixed line-heights below 1.4; headings 1.2 acceptable (SC exception).
  - **2.1.1 Keyboard / 2.1.2 No Keyboard Trap:** all actions keyboard-reachable, no traps.
  - **2.4.7 Focus Visible:** ring pattern above.
  - **2.5.5 Target Size (AAA target, AA minimum 24×24):** all buttons ≥ 44 px tall at 320 px; icon-only controls do not exist in v1.
  - **3.3.1 Error Identification:** error state is both visual (oxblood tick) and text; announced via ARIA live.
  - **4.1.3 Status Messages:** state transitions go through `role="status"` live region.

- **Serif-at-small-size legibility guard:**
  - Body serif (Source Serif 4) never below 15 px.
  - Italic serif in `HintRule` and caption line 1 never below 13 px.
  - Sans (Inter) is used for metadata small-caps (11 px) and button labels on quiet buttons — small-caps sans is demonstrably more legible at that size than italic serif.
  - If QA testing at 320 px on a low-DPI display shows the serif body is subjectively hard to read, the fallback is a single Tailwind token swap: `font-serif` → `font-sans` for body while keeping the card/frame composition intact. This is called out in § 6 as OQ-1.

- **Focus pattern summary:** `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-stone-900 focus-visible:ring-offset-2 focus-visible:ring-offset-[#FAF7F0]` applied uniformly. No `:focus` without `:focus-visible` — mouse users don't see rings.

- **Keyboard map:**
  | Key | S-IDLE | S-CAPTURING | S-READY | S-ERROR |
  |---|---|---|---|---|
  | Tab | Capture | Capture (disabled, skipped) | Capture → Copy → Download | Retry → Capture |
  | Shift-Tab | reverse | reverse | reverse | reverse |
  | Enter/Space | Capture fires | no-op | focused action fires | focused action fires |
  | Esc | no-op | no-op | clear to S-IDLE (optional) | clear to S-IDLE |

## 6. Open questions and handoff

- **Open questions:**
  - **OQ-1 (legibility):** Validate Source Serif 4 at 15 px body on the lowest-end realistic editor display (Windows 125 % scaling, 1080p). If perceived fuzziness is reported, swap body font to Inter and keep serif only for the masthead, plate label, caption line 1–2, and hint. Decision owner: UI Designer + one content-editor user test.
  - **OQ-2 (plate numeral):** v1 is always "Plate I" because no history is stored (ADR-0006). If we ever persist the last N captures, plate numerals become meaningful (I, II, III…). Until then: keep "Plate I" or drop the numeral entirely? Recommendation: keep — it establishes the archival register even with a single plate.
  - **OQ-3 (dimensions source):** The PRD does not guarantee the `/screenshot` endpoint returns dimensions. If absent, caption line 3 omits the `× HEIGHT` half gracefully. Confirm with the server route contract during implementation.
  - **OQ-4 (dark mode follow-on):** Does Sitecore Pages ever expose an editor-theme signal the iframe could read? If yes, a dark-mode "Dossier" variant of Passepartout becomes a future possibility. Not in v1.
  - **OQ-5 (font delivery):** Ship Source Serif 4 and Inter via `next/font` (local) to avoid runtime Google Fonts CSP complications in the Marketplace iframe (NFR-Co-01). Confirm CSP allows the `next/font`-generated style tag.
  - **OQ-6 (copy micro-UX):** Should the copy confirmation be inside the frame (over the image) or below the Copy button as an inline message? This spec says below-the-button. Revisit after clickdummy review.
