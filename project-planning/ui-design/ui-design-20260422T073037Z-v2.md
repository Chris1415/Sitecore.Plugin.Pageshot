## 1. Design context & assumptions

### Audience

- **Primary:** Content editors and marketers inside SitecoreAI Pages. They screenshot pages several times a day for Slack/Teams review threads. They are not power users; they respond to affordances that feel like consumer apps (camera, Instagram, Loom) more than to enterprise dashboards.
- **Secondary:** Solution engineers demoing pages, and QA reviewers attaching shots to bug reports. Both tolerate more chrome than marketers but also prefer tools that feel nice over tools that feel austere.
- **Context of use:** a narrow right-rail Context Panel embedded as an iframe inside the Sitecore Pages editor. Panel is the only surface. Sessions are short — open panel, capture, copy or download, move on.

### Goals and success criteria

- **G1 — Make the capture action feel like the hero.** In a narrow column the capture button is 90 % of the first impression. It should read as "press this" from across the room and feel satisfying to activate.
- **G2 — Preview that rewards the user.** After capture, the image gets a cinematic presentation — a card with soft shadow, a clear frame — not a naked `<img>` stuffed under a heading. The preview is the payoff.
- **G3 — Copy and Download are first-class but secondary.** Paired pill actions, visually subordinate to the preview but unambiguously the next step.
- **G4 — Errors stay friendly.** Error UI uses the same card + icon language as success. No red alert bars. An error is just a different "card" — a "didn't quite catch that" moment.
- **G5 — Meet PRD § 6 acceptance criteria.** All AC-1..AC-5 honored in this variant's visual language (inline hint, elapsed-time counter, retry path, copied-toast, filename template).

### Constraints (brand, a11y, stack)

- **Stack:** Next.js 15 App Router + React 19 + TypeScript + Tailwind CSS per ADR-0004. Tokens must be expressible in Tailwind utility classes (or a small `tailwind.config` extension). Radix + shadcn/ui primitives are available and used selectively (Button, Toast, Tooltip, VisuallyHidden).
- **Viewport:** 320–400 px wide panel, vertical scroll only. No horizontal scroll at 320 px. This is a hard floor.
- **Iframe host:** Sitecore Pages editor (light-themed chrome around us). We pick our own scheme — **warm light with a single warm accent**, working in both the host's light chrome and standalone testing.
- **No persistence** (ADR-0006). Everything visual is driven by React state. No "recent captures" shelf.
- **Accessibility:** WCAG 2.1 AA; full keyboard operability; visible focus rings; ARIA live regions for state changes; reduced-motion respected.
- **Server-side state** (ADR-0002). The client never holds credentials — no auth UI needed.

### Reference / inspiration

Grounded in real consumer-capture patterns:

- **CleanShot X** — the quick-access overlay after capture, the calm card-based presentation of the just-taken shot with annotation/action affordances below it. Not the menu-bar-app parts; the post-capture card.
  - [CleanShot X](https://cleanshot.com/) · [CleanShot X Features](https://cleanshot.com/features) · [CleanShot X Review (The Sweet Bits)](https://thesweetbits.com/tools/cleanshot-review/)
- **Loom Chrome extension** — the floating red circular capture control; the idea that a recording/capture button should be unambiguously THE element on screen. We take the hero-button principle, not the color.
  - [Loom Screen Recorder](https://www.loom.com/screen-recorder) · [Loom Chrome Screen Recorder](https://www.loom.com/products/chrome-screen-recorder)
- **iOS Camera / camera-app UI patterns** — big circular shutter, viewfinder framing, haptic-ish feedback via scale and ring animation. Shutter should pulse-shrink on press and settle. The "flash of light" idea is adapted here as a soft bloom overlay when the capture resolves.
  - [Design Apps with Beautiful Camera Control (LeewayHertz)](https://www.leewayhertz.com/design-apps-with-a-great-camera-user-experience/) · [UI Design a Camera App in Figma (Design+Code)](https://designcode.io/ui-camera/)
- **Polaroid / instant-camera metaphor** — the preview lives in a paper-like card with a visible bottom ledge where metadata (site, page, timestamp) sits. Tactile, warm, and it solves a real layout problem (where to put "shows last saved version").
  - [Warm neutral palettes 2025 (Davey & Krista)](https://daveyandkrista.com/top-trending-color-palettes-for-websites/) · [Figma Color Combinations](https://www.figma.com/resource-library/color-combinations/)
- **Raycast extension UI** — native-feeling density, React-driven, opinionated layout as a canvas. Reminds us that "app-like" inside a shell can still feel at home.
  - [Raycast User Interface API](https://developers.raycast.com/api-reference/user-interface)

### Assumptions made

- Panel mounts already know `pageId`, `siteName`, `pageName` (from `pages.context`). We do not design a "no page context" skeleton in v1.
- `navigator.clipboard.write` is available in target browsers (NFR-Br-01). Fallback copy-error handling uses the same error-card pattern (routed through `AC-2.4`).
- The panel is always mounted at the Context Panel width — we do not design a "wide" mode.
- Sitecore Pages editor does not inject parent-document CSS. Our Tailwind layer is the only source of styles.
- User can tolerate ~10 s capture latency; elapsed-time counter (after 5 s) is the patience-builder, not a cancellation UI.

## 2. Design direction

### Visual direction — **Shutterbug**

**Shutterbug** is a warm, tactile, camera-app-inspired direction: an amber shutter button is the hero, and captures land in a paper-polaroid card that sits on a cream surface. The panel feels like a tiny desk — a soft background, one big beautiful button, and a photo that "arrived" on it. Low chrome, high character. It does not apologize for being a screenshot tool; it makes the screenshot the event.

**Aesthetic rationale (one paragraph).** Marketers do this dozens of times a day. Enterprise tools usually treat that as "make it dense and invisible." Shutterbug inverts that — if you're going to do a thing repeatedly, it should feel good every time. The shutter button borrows from iOS/camera-app convention because that muscle memory is universal; the polaroid framing borrows from CleanShot X's "your capture is a thing, not a data blob"; the warm-cream palette borrows from the 2025 Notion-adjacent reaction against cold corporate white. The design is opinionated — but the opinion is "your five-second errand should be pleasant," which is a defensible stance for a utility that runs in a panel you see constantly.

**North-star feelings:** *ready*, *warm*, *satisfying*, *tactile*, *brief*.
**Anti-patterns to avoid:** enterprise grey, form-dense, red error banners, full-width rectangular CTAs, "professional" cold blues.

### Layout and structure

The panel is a single vertical column with four zones. At 320 px width the column is `px-4`; at 360 px it's `px-5`; at 400 px it's `px-6`. Vertical rhythm uses an 8-px scale (Tailwind default).

```
┌────────────────────────────────────────┐
│ ◉ PageShot                         · · │ ← header (44 px)
│                                        │
│   "Ready when you are."                │ ← subtitle / status line
│   Shows the last saved version.        │
│                                        │
│                                        │
│              ╭───────╮                 │
│              │       │                 │
│              │  📷   │  ← shutter      │ hero zone (~200 px)
│              │       │                 │
│              ╰───────╯                 │
│              Capture                   │ shutter label
│                                        │
├────────────────────────────────────────┤
│ ┌────────────────────────────────────┐ │
│ │ ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ │ │
│ │ ░░░░░ (preview or empty) ░░░░░░░░ │ │ polaroid card
│ │ ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ │ │ (only in ready/
│ │                                    │ │  error states)
│ │ acme / home · 2026-04-22 09:14     │ │ paper ledge
│ └────────────────────────────────────┘ │
│                                        │
│  ┌──────────┐  ┌──────────────┐        │
│  │   Copy   │  │   Download   │        │ action pills
│  └──────────┘  └──────────────┘        │
└────────────────────────────────────────┘
```

**Zones, top to bottom:**

1. **Header** — 44 px. Wordmark left; nothing on the right in v1 (reserved for a future overflow menu).
2. **Status line / hint** — two lines of text. First line is state-specific ("Ready when you are." / "Catching it…" / "Got it." / "Didn't quite catch that."). Second line is always the FR-12 hint.
3. **Hero shutter** — a 112 × 112 px circular button, generously spaced. This is the center of gravity of the panel. Label "Capture" sits directly below in Geist Sans 500 13 px.
4. **Polaroid card** — appears in ready and error states. White surface, `rounded-3xl`, `shadow-xl` with warm amber-tinted shadow, 8 px of white "frame" padding around the image, and a paper-colored ledge at the bottom (14 px tall) containing site/page/timestamp in Geist Mono 11 px.
5. **Action pills** — `Copy` and `Download`, side by side, `rounded-full`, `h-10`. Copy is slightly smaller (less text). Disabled in idle/capturing states.

**Idle state** hides the polaroid entirely and instead shows a soft dashed placeholder directly below the hint ("Tap Capture to catch this page."). This keeps the panel from feeling top-heavy when empty.

### Key components and patterns

- **Shutter button** — the single most important element. Amber-500 fill, white camera glyph, amber-300 outer ring with 4 px gap (the "aperture"). Pressed state scales to 0.92 with a 120 ms spring. Disabled state drops to amber-200 and loses the inner ring. Capturing state replaces the camera glyph with a spinning aperture ring + center dot.
- **Polaroid card** — white surface over cream background, visible because of shadow and subtle border (`border border-stone-200/60`). Bottom ledge uses a 1 px top-border of the same tone. The ledge is where the hint lives in success state and where the filename lives in a tiny mono font.
- **Action pill** — full-rounded, stone-900 text on white, 1 px stone-300 border in default. Hover: amber-50 background. Pressed: amber-100. Disabled: 50 % opacity, no pointer events. `Copy` becomes `Copied ✓` for 1.8 s after a successful copy (stays as the same pill — no separate toast component for this variant; the in-situ confirmation matches the camera-app feel).
- **Elapsed-time pill** — appears inside the shutter's label slot after 5 s. "Still catching… 6 s" in Geist Mono 12 px, amber-700. Updates every 1 s. ARIA-live polite.
- **Error card** — same polaroid frame, but the image slot holds a large icon (AlertCircle, WifiOff, or similar) on a stone-50 background, with the error title in the ledge area. A retry pill replaces the Download pill for errors.
- **Copied toast (fallback)** — for `AC-2.3`, since the pill already animates, there is no separate toast; however an `aria-live="polite"` sr-only node announces "Copied to clipboard" for assistive tech.

## 3. Key screens and flows

The extension has one screen — the panel — with five user-visible states. Each is effectively a "screen" for design purposes.

### Screens (panel states)

| # | State | Purpose | Distinguishing elements |
|---|---|---|---|
| S1 | **Idle** | Panel just loaded, no capture yet | Shutter active, no polaroid; dashed placeholder where polaroid would go; Copy/Download disabled |
| S2 | **Capturing (< 5 s)** | Capture in flight, fresh | Shutter shows spinner (aperture-ring rotation), label "Catching it…"; no elapsed counter yet |
| S3 | **Capturing (≥ 5 s)** | Capture slow, reassure user | Same as S2 + "Still catching… 6 s" mono pill under the shutter, updating every second |
| S4 | **Ready** | Capture succeeded | Polaroid card with image; hint moves to the ledge alongside filename; Copy + Download enabled; shutter re-enabled (re-capture replaces) |
| S5 | **Error** | Capture failed (any code) | Polaroid card with icon + title instead of image; Retry pill replaces Download; Copy disabled; inline message with a specific actionable sentence |

Sub-states of S4 / S5:
- **S4a — Just-copied** — Copy pill morphs to `Copied ✓` amber, auto-reverts after 1.8 s.
- **S4b — Clipboard-denied** — visually identical to S5 but the message is the clipboard-denied variant and the action pills show Download + disabled Copy.
- **S5a–S5d** — four error codes (`auth`, `not_found`, `upstream_unavailable`, `network`) with per-code icon + title + subtitle per AC-5.1..AC-5.4.

### Critical user flows

**Flow A — Happy path (AC-1.1..1.5, AC-2.1..2.3, AC-3.1..3.4):**

1. Panel mounts → **S1 Idle**. Shutter focused (can press Enter immediately).
2. User presses Shutter → press animation (scale 0.92) → **S2 Capturing**.
3. At 5 s, transitions to **S3 Capturing (elapsed)**. Counter ticks every 1 s.
4. Route handler returns `{ ok: true, image }` → soft bloom overlay (200 ms) → **S4 Ready**. Polaroid slides up + fades in. Focus moves to Copy pill.
5. User presses Copy → blob copied → pill morphs to `Copied ✓` (S4a) for 1.8 s → reverts.
6. User presses Download → file downloads with the PRD § 5.6 filename → pill briefly shows a checkmark, reverts. (No state change.)
7. User presses Shutter again → **S2** replaces the image on success (FR-11).

**Flow B — 404 (AC-4.3, AC-5.2):**

1. S1 → press → S2 → S5 "not_found".
2. Polaroid shows a "page not found" illustration (simple line art of a page with a dashed outline); title: "We couldn't find that page."; subtitle: "Save it first, then try again."; Retry pill (disabled until the user has dismissed and re-pressed Shutter is implicit — Retry IS the shutter-press).
3. Press Retry → S2.

**Flow C — 5xx / timeout / network (AC-5.3, AC-5.4):**

1. S1 → press → S2 → S3 → S5 "upstream_unavailable" or "network".
2. Polaroid shows a WifiOff (network) or generic AlertCircle (upstream) icon; title is the code-specific sentence; Retry pill is the primary action.
3. Press Retry → S2.

**Flow D — Clipboard denied (AC-2.4):**

1. In S4, user presses Copy. `navigator.clipboard.write` rejects.
2. Copy pill briefly shakes (80 ms translate-x) — reduced-motion: no shake.
3. A small inline message appears above the action pills: "Clipboard access was blocked. Use Download instead." with `role="status"` + `aria-live="polite"`.
4. Copy pill goes to disabled. Download pill remains primary.

**Flow E — Keyboard-only (NFR-A-01):**

1. Focus order at mount: Shutter → Copy → Download (skip disabled). 
2. Shutter activates on Enter or Space.
3. After capture resolves, focus auto-advances to Copy (single `focus()` call on the pill).
4. After Copy, focus stays on Copy so the user can Tab to Download next.
5. `Esc` from anywhere returns focus to Shutter. (No clear-to-idle in v1 keyboard map; Shutter re-press replaces content.)

### Responsiveness

The panel has a single break-axis (width). We design for three widths:

| Width | Behavior |
|---|---|
| **320 px (hard floor)** | Container `px-4`. Shutter stays 112 × 112 px — it is the one element that does NOT shrink. Action pills go full-width stacked vertically (Copy on top, Download below) because two pills side-by-side at 320 px – 32 px padding = 288 px usable cannot accommodate both comfortably with `Download` (9 chars). Polaroid card fills 100 % width minus padding. Ledge text uses `text-[10px]`. |
| **360 px (default target)** | Container `px-5`. Action pills go side-by-side with 8 px gap; Copy is `flex-none` width ~96 px, Download takes remaining space. Polaroid fills width. Ledge uses `text-[11px]`. |
| **400 px (comfortable)** | Container `px-6`. Same as 360, with more generous vertical spacing in the hero zone (`py-10` instead of `py-6`) and a larger shutter label (14 px vs 13 px). Polaroid gets `p-3` frame padding instead of `p-2`. |

No `@media` queries are needed for width-responsive changes because the panel width is stable once mounted. We use container-query-equivalent breakpoints via a wrapper with `@container` or, failing that, a small `useResizeObserver` hook that sets a `size` variant on the root. Decision: use Tailwind's `@container` queries — supported in Tailwind 4, aligns with ADR-0004's "stay close to the framework default."

Vertical: the panel can be any height ≥ ~600 px. Below that, everything below the shutter scrolls — the shutter itself should stay sticky-visible for quick re-capture:

- **Sticky Shutter (optional-nice):** at panel heights < 600 px, the shutter zone becomes `sticky top-0` with a soft amber-50 gradient backdrop so that scrolling the preview never loses the primary action. This is a v1 target; if it costs > 2 hours of implementation it's dropped to v1.5.

#### PolaroidCard (ready)

- Frame: `bg-white rounded-3xl p-2 border border-stone-200/60 shadow-xl shadow-amber-900/10`
- Image slot: `rounded-2xl overflow-hidden aspect-[4/3] bg-stone-100` — image fills width, `object-cover object-top` so the page's top is always visible.
- Ledge: `px-3 py-2 mt-1 border-t border-stone-200/60 text-[11px] font-mono text-stone-600 flex items-center justify-between gap-2` — left shows `{siteName}/{pageName}`, right shows `HH:mm` (local), both truncated with `truncate`.

- Image slot: `aspect-[4/3] bg-stone-50 flex flex-col items-center justify-center gap-2` — large icon (32 px, stroke 1.5) in `text-rose-500`, title in `text-sm font-medium text-stone-900`, subtitle in `text-xs text-stone-600 text-balance text-center px-4`.
- Ledge: shows the FR-12 hint "Shows the last saved version of this page." in `text-[11px] text-stone-500` — italic, center-aligned, no filename.

| State | Tailwind intent |
|---|---|

#### EmptyPreview (idle)

- `rounded-3xl border-2 border-dashed border-stone-300 bg-white/40 py-10 px-4 text-center text-sm text-stone-500`
- Content: "Tap Capture to catch this page." (13 px Geist Sans).
- No illustration — keeps idle visually lighter than ready so the shutter reads as the thing to do.

#### Soft-bloom overlay (transition S3 → S4)

- A full-panel fixed overlay: `absolute inset-0 pointer-events-none bg-[radial-gradient(circle_at_center,white_0%,transparent_70%)] opacity-0`
- Animates `opacity-0 → opacity-90 → opacity-0` over 200 ms on capture success. Reduced motion: skipped entirely.

### 4.4 Tailwind token intent (summary)

Added to `tailwind.config.ts` theme extensions (all using Tailwind defaults otherwise):

```ts
// Conceptual — implementer resolves exact values
theme: {
  extend: {
    colors: {
      // reuses Tailwind's amber and stone palettes unchanged
    },
    boxShadow: {
      shutter: "0 10px 30px -8px rgb(245 158 11 / 0.35)",
      polaroid: "0 20px 40px -12px rgb(120 53 15 / 0.15)",
    },
    keyframes: {
      shake: { "0%,100%": { transform: "translateX(0)" }, "50%": { transform: "translateX(-4px)" } },
      bloom: { "0%,100%": { opacity: "0" }, "50%": { opacity: "0.9" } },
    },
    animation: {
      "shutter-bloom": "bloom 200ms ease-out",
    },
  },
},
```

No custom palette beyond Tailwind's `amber`, `stone`, and `rose` families. This is deliberate — keeps the tokens trivially auditable and avoids a bespoke design-system burden for a one-panel app.

### 4.5 Accessibility notes

- All interactive elements are `<button>` (not divs). Shutter carries `aria-label="Capture screenshot"`; when capturing, `aria-busy="true"` + `aria-label="Capturing screenshot"`.
- A single `<div role="status" aria-live="polite" class="sr-only">` at panel root receives announcements on every state change: "Capturing started", "Still capturing, 6 seconds", "Screenshot ready", "Copied to clipboard", "Capture failed: [title]. [subtitle]".
- Focus ring is always visible under `:focus-visible` — never removed. Ring is amber-400 (≥ 4.5:1 against amber-50 background).
- Text contrast verified:
  - stone-900 on amber-50 → 16.6:1 (AAA)
  - stone-600 on amber-50 → 6.1:1 (AA large + small)
  - white on amber-500 → 3.4:1 — **fails 4.5:1 for body text but passes 3:1 for large UI components** (button text at 14 px/500 is on the borderline). **Resolution:** shutter's internal glyph is an icon (exempt), and the label "Capture" is outside the button in stone-900. No body text sits on amber-500.
  - amber-700 on amber-50 → 7.2:1 (elapsed-time pill)
  - rose-500 on white → 4.3:1 — **just under 4.5**. **Resolution:** error title uses `text-rose-600` (5.7:1), the icon uses rose-500 (decorative, contrast exempt).
- Reduced motion: all spring and bloom animations collapse to opacity or are removed; no essential information conveyed by motion.
- The polaroid's captured image is `<img alt="Screenshot of page {pageName} on {siteName}, captured {capturedAt}">`.

## 5. Responsiveness and accessibility

### Breakpoints

Container-query-based (via Tailwind `@container` on `PanelRoot`):

| Name | Min width | Behavior |
|---|---|---|
| `xs` (hard floor) | 320 px | Stacked action pills (Copy over Download); `px-4`; shutter label 13 px |
| `sm` (default target) | 360 px | Side-by-side pills; `px-5`; shutter label 13 px |
| `md` (comfortable) | 400 px | Side-by-side pills, looser vertical rhythm (`py-10` on hero); `px-6`; shutter label 14 px |

No `md`-and-up layouts — above 400 px we render `md` behavior unchanged; the Context Panel does not get wider than ~420 px in practice.

**Vertical overflow** is always `overflow-y-auto` on the root; horizontal is `overflow-x-hidden`. No zoom-out gestures assumed — the panel is a browser iframe in Sitecore Pages.

### A11y requirements (WCAG 2.1 AA)

| Requirement | How we meet it |
|---|---|
| 1.4.3 Contrast (Minimum) | All text ≥ 4.5:1 against its background; large text ≥ 3:1 (see § 4.5). |
| 1.4.11 Non-text Contrast | Shutter border/ring, pill borders, focus rings all ≥ 3:1. |
| 2.1.1 Keyboard | Full keyboard flow: Tab, Shift+Tab, Enter, Space, Esc (see Flow E). |
| 2.4.3 Focus Order | DOM order = visual order (header → status → shutter → pills). |
| 2.4.7 Focus Visible | `:focus-visible` never suppressed. Amber-400 ring, 2 px offset. |
| 3.3.1 Error Identification | Error state shows inline error card with icon + title + subtitle, plus LiveRegion announcement. |
| 3.3.3 Error Suggestion | Each error code's subtitle tells the user the specific next step (AC-5.1..5.4). |
| 4.1.2 Name, Role, Value | All buttons have accessible names; `aria-busy` on shutter during capture; `aria-live` on state-announce region. |
| 4.1.3 Status Messages | `role="status" aria-live="polite"` region present and updated. |

### Keyboard map

| Key | Action |
|---|---|
| Tab | Advance focus: Shutter → Copy → Download (or Retry) |
| Shift+Tab | Reverse |
| Enter / Space | Activate focused button |
| Esc | Return focus to Shutter (from any inner element) |

### Focus strategy

- On panel mount → focus Shutter.
- On S2 → S4 transition → focus Copy (so keyboard users don't have to chase the preview).
- On S2 → S5 transition → focus the Retry pill.
- On re-capture press from S4 → focus stays on Shutter through S2 → returns to Copy on S4.

## 6. Open questions and handoff

### Open questions (developer-ready)

1. **Sticky shutter at short panel heights** — is panel-height < 600 px realistic in Sitecore Pages? If yes, implement sticky. If it adds > 2 hours, drop.
2. **Geist font delivery** — use `next/font/google` for Geist Sans + Mono, or bundle via `geist` npm package? Either works; `geist` package is the official route and self-hosts the font.
3. **Sound on capture?** A 20 ms shutter-click sound would complete the camera metaphor, but panels in iframes should probably not make noise without user opt-in. **Recommendation: no sound in v1.** Flag for a follow-up if editors ask for it.
4. **Icon weight** — stroke-width 2 (Lucide default) vs 1.5 (softer). **Recommendation: 1.5 for error-card icons, 2 for action-pill icons.** Engineer's call if they disagree after implementation.
5. **Filename preview in the ledge** — we currently show `{site}/{page} · HH:mm`. Should the ledge also expose the full download filename on hover (tooltip)? PRD doesn't require it but it would help power users verify filename format. **Recommendation: add a Radix Tooltip on the ledge, opt-in.**
6. **Elapsed-time counter format** — "Still catching… 6 s" vs "Still catching… 00:06". **Recommendation: the former** — more human, matches "Shutterbug" personality. Localization is out of scope (PRD is EN-only in v1).
7. **Host theme reconciliation** — if Sitecore Pages ever ships a dark theme, does our warm-cream panel jar? **Recommendation: accept this as a v2 concern, ship light-only.** Record as a tech-debt note in the ship report.
