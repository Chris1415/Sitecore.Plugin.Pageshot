## 1. Design context & assumptions

### Audience

Per PRD § 4, primary users are **content editors and marketers** working in SitecoreAI Pages who share in-progress pages with colleagues. Secondary are **solution engineers / consultants** giving live demos and **QA reviewers** attaching screenshots to bug reports.

This variant leans deliberately toward the *secondary* audience's mental model — the SE/consultant and the QA reviewer who already live in dev-tools panels and keyboard-driven workflows. The bet: even the primary "marketer" audience, when working inside Sitecore Pages, is already in a **tool-like** context panel next to component trees, field inspectors, and layout sliders. A panel that feels like the rest of the editor's pro-tool surface is legible at a glance; a panel that feels playful or decorative competes with the content they're actually editing.

### Goals and success criteria

Tied to PRD § 3:

- **G1 — Reduce time-to-shared-screenshot.** The UI must make Capture → Copy/Download a two-click, glanceable path. Power-user keyboard operation (`C` to capture, `Y` to copy, `D` to download) should feel native, not bolted on.
- **G2 — Eliminate editor-chrome noise from shared artefacts.** The preview panel renders the returned image on a neutral graphite surface so the user can verify chrome-free output before copy/download.
- **G3 — Graceful behavior under real-world API conditions.** Every state (idle / capturing / ready / error) has a recognizable visual signature so status is readable without reading labels. Metadata (elapsed time, capture time, filename preview) is always visible when relevant.

### Constraints

- **Extension surface:** Sitecore Pages Page Builder **Context Panel**, narrow vertical column.
- **Viewport:** 320–400 px wide (hard floor 320 px), scrollable vertically, no horizontal scroll.
- **Host:** HTTPS iframe loaded by Sitecore Pages (light-themed editor).
- **Stack (ADR-0004):** Next.js 15 App Router, React 19, TypeScript strict, **Tailwind CSS** only. Radix/shadcn primitives permitted. Design tokens must be expressible in Tailwind (arbitrary values allowed, but prefer scale values).
- **Architecture (ADR-0002, ADR-0006):** Client is stateless beyond React state; no secrets client-side; no persisted images or history.
- **Accessibility:** WCAG 2.1 AA. Full keyboard operability. Visible focus rings. ARIA live regions announce state changes.
- **States:** idle, capturing (spinner + elapsed counter after 5 s), ready (preview + Copy/Download), error (message + Retry). Per PRD § 11 and FR-10.

### Reference / inspiration

Synthesized from real pro-tool surfaces, not copied:

- **Linear's inspector panels** — tight Major Second typographic scale (1.125), neutral grayscale with a single accent, generous use of monospace for identifiers. See the `Sohne` + `Sohne Mono` pairing Linear uses. [Linear design — LogRocket](https://blog.logrocket.com/ux-design/linear-design/)
- **Raycast extension UI** — `ActionPanel` + keyboard shortcut affordances shown directly next to actions (`⌘C`, `⌘↵`). Shortcut glyphs are part of the visual language, not a help-menu afterthought. [Raycast API — User Interface](https://developers.raycast.com/api-reference/user-interface), [Raycast — Keyboard](https://developers.raycast.com/api-reference/keyboard)
- **Vercel's 2026 dashboard redesign** — dense sidebars, developer-centric rhythm, collapsible rails. [Vercel dashboard redesign](https://vercel.com/blog/dashboard-redesign), [Vercel — new dashboard](https://vercel.com/try/new-dashboard)
- **shadcn/ui Sidebar + Radix primitives** — accessible keyboard navigation, focus traps, and resize patterns for narrow sidebars. [shadcn/ui sidebar](https://ui.shadcn.com/docs/components/radix/sidebar)
- **Dense-information UI typography** — Major Second scale (1.125) for compact dashboards; justification for tight vertical rhythm in the panel. [Design Systems — typography guides](https://www.designsystems.com/typography-guides/)

### Answers to clarifying questions

The user's brief was "designer's call on everything except divergence." Resolved Q&A:

| Q | Resolution | Rationale |
|---|---|---|
| Dark vs light mode? | **Dark panel** (graphite, not pitch black). Background `#0F1115`. | The host editor is light-themed; a graphite panel reads as a distinct "inspector tool" rather than competing editor chrome. A near-black (not true-black) avoids the harsh jarring seam in the iframe. Committed — this is the variant's core stance. |
| Primary typeface? | **Inter** (body/UI), **JetBrains Mono** (identifiers, metadata, shortcuts). | Inter is freely licensable, weight-rich, renders cleanly at 12–14 px. JetBrains Mono has excellent small-size legibility and character distinction, now a de-facto standard for developer tooling. |

Amber gives the panel a distinct "indicator LED / caution light" signature that reads as instrumentation, not a generic dashboard. Used sparingly: primary button, elapsed-time counter, focus rings. |
| Motion vocabulary? | Minimal, mechanical. Spinners rotate at 1 rev/s; elapsed counter ticks once per second with no ease; state transitions 120 ms linear. Respects `prefers-reduced-motion`. | Utility tone. Motion is signal, not polish. |
| Dense vs airy spacing? | **Dense.** 4 px base grid, major rhythms at 8/12/16. Panel inner gutter 12 px. | Matches the utility-inspector lens. PRD § 11 layout sketch already implies tight vertical stacking. |
| Show keyboard shortcuts? | **Yes, inline** — rendered as kbd glyphs on every primary action. | Core to the pro-tool lens. A marketer who never uses them loses nothing; a power user gets immediate muscle memory. |
| Show capture metadata? | **Yes, always visible in ready state:** filename preview, capture timestamp, image dimensions. | PRD does not require it, but it's the fingerprint of a pro tool. Gives QA reviewers the verification they need before copy/download. |
| Brand mark? | Small wordmark "PageShot" top-left in Inter semibold tracking-tight, with a tiny camera-aperture glyph. No logo assets; glyph is a Heroicon/Lucide `aperture` at 14 px. | Marketplace apps need identity; a tiny mark is enough. |

### Assumptions made

1. **The panel is always 320–400 px wide.** I treat 320 px as the design target and 400 px as a "breathing room" ceiling. No design decisions require > 400 px.
2. **Tailwind v4 + arbitrary values are allowed** — token values that don't map to the default scale (e.g., `bg-[#0F1115]`) are expressible and permitted.
3. **Lucide icons** are the icon set (matches shadcn/ui conventions). Specifically: `Aperture`, `Camera`, `Copy`, `Download`, `Check`, `AlertTriangle`, `RefreshCw`, `Loader2`.
4. **The hint "Screenshot shows the last saved version of this page." (FR-12)** is rendered as a persistent sub-label under the Capture button, not a toast, so it survives state transitions.
5. **Previous capture's image is cleared on new capture start** (FR-11), but the panel briefly shows the previous image during the capturing state to avoid a blank flash — it's dimmed to 30 % opacity with the spinner overlay on top. Small UX nicety; does not violate FR-11's "not retained" requirement because it's only shown during the in-flight request and replaced on success/error.
6. **Mock data for preview examples:** `siteName="acme-marketing"`, `pageName="home"`, `capturedAt=2026-04-22T10:14:00Z`, image dimensions `1440×900`.

## 2. Design direction

### Visual direction — "Graphite Console"

**One-paragraph aesthetic rationale:**
Graphite Console is a dark, dense inspector surface that reads as a piece of instrumentation next to the Pages editor rather than a decorative widget. The panel is near-black graphite (`#0F1115`) with a single warm amber accent (`#F59E0B`) that behaves like an indicator LED — lit when a primary action is available, pulsing softly when the capture is in flight. Typography is a two-typeface system: Inter for UI body text and JetBrains Mono for every identifier, metadata field, keyboard shortcut, timestamp, and filename — monospace is *informational*, not decorative. Spacing is tight (4 px base, 12 px panel gutter), rules are hairline `border-white/8`, and interactive surfaces shift by one step on hover (`white/[0.04]` → `white/[0.08]`). The effect is Linear-inspector-meets-Raycast-extension: a tool that shows you exactly what it knows, presents the shortcut next to every action, and never tries to be cute.

### Layout and structure

Vertical stack with no horizontal overflow. Panel inner gutter 12 px left/right, 12 px top/bottom. Sections are separated by hairline dividers (1 px `border-white/8`), not margin stacks.

```
┌────────────────────────────────────────┐  ← 320 px wide
│ ◎ PageShot                   v1 · β   │  header row (28 px)
├────────────────────────────────────────┤
│ CONTEXT                                │  section label (10 px mono uppercase)
│ site      acme-marketing               │  metadata rows (mono, label/value)
│ page      home                         │
│ id        7f3a·9e2b                    │  truncated mono pageId
├────────────────────────────────────────┤
│ [ ◉ Capture screenshot         C ]    │  primary action, kbd glyph right
│  Shows the last saved version          │  hint sub-label
├────────────────────────────────────────┤
│ PREVIEW                                │  section label
│ ┌──────────────────────────────────┐  │  preview frame (16:10, dotted border
│ │                                  │  │   in idle; image in ready)
│ │   Empty · ready to capture       │  │
│ │                                  │  │
│ └──────────────────────────────────┘  │
│ —                                      │  metadata rows appear in ready state
├────────────────────────────────────────┤
│ [ ⧉ Copy     Y ] [ ⬇ Download   D ]   │  secondary actions, disabled in idle
├────────────────────────────────────────┤
│ Shift+? for help                       │  footer hint (10 px mono, muted)
└────────────────────────────────────────┘
```

Section hierarchy (top to bottom):

1. **Header** — wordmark + version badge. 28 px tall.
2. **Context block** — section label `CONTEXT`, three metadata rows (site, page, id). Always present.
3. **Primary action** — Capture button + persistent hint.
4. **Preview block** — section label `PREVIEW`, the frame (state-dependent), and a metadata row (state-dependent: filename + dimensions + capture time when ready; elapsed counter when capturing; error code when error).
5. **Secondary actions** — Copy and Download side-by-side (2-column grid). Disabled until ready. kbd glyph on each.
6. **Footer** — muted "Shift+? for help" hint.

### Key components and patterns

- **SectionLabel** — 10 px JetBrains Mono, uppercase, `text-neutral-500`, letter-spacing wide. Appears above each block.
- **MetadataRow** — label/value pair; label 11 px mono `text-neutral-500`, value 12 px mono `text-neutral-200`, truncating with middle-ellipsis for IDs.
- **PrimaryButton** — full-width, 36 px tall, graphite base with amber left-border indicator when enabled, kbd glyph on the right edge.
- **SecondaryButton** — half-width (in 2-col grid), 32 px tall, no accent border, kbd glyph on the right.
- **PreviewFrame** — fixed 16:10 aspect, dashed `border-white/10` in idle/error, solid `border-white/12` in ready. Background `bg-black/40`.
- **KbdGlyph** — small mono glyph in a 16×16 tile with `border-white/10`, `bg-white/[0.04]`, `text-neutral-300`. Appears on the right edge of every primary/secondary action.
- **ElapsedCounter** — mono, amber, in the preview metadata row when capturing > 5 s.
- **StatusPill** — used for error state; icon + short code in a small pill (`bg-amber-500/10 text-amber-300 border-amber-500/20`).
- **Toast/Inline confirmation** — "Copied" confirmation appears as an inline pill under the Copy button for ≤ 2 s per AC-2.3.

## 3. Key screens and flows

The PageShot panel is **one screen** with **five panel states**. Each state below is rendered in the same vertical shell described in § 2; only the Capture button, Preview block, secondary action row, and live-region text change.

### State 1 — Idle (first render)

- Header: wordmark + version badge.
- Context: all three metadata rows populated from `pages.context`.
- Primary: Capture button **enabled**, amber left-border lit.
- Hint: "Shows the last saved version."
- Preview: dashed frame, centered muted text "Empty · ready to capture". `—` metadata row.
- Secondary: Copy and Download both **disabled** (opacity 40 %, cursor not-allowed).
- ARIA live: `polite` region says nothing (initial mount).

### State 2 — Capturing (in flight, < 5 s)

- Capture button: **disabled**, replaced label becomes `◐ Capturing…` with a rotating spinner glyph in the left-border slot (amber).
- Hint: unchanged ("Shows the last saved version.").
- Preview: frame shows previous image dimmed to 30 % (or "Capturing the current page…" muted label if no previous), with a centered `Loader2` spinner at 24 px. Metadata row shows `0.2 s` ticking.
- Secondary: Copy and Download **disabled**.
- ARIA live (polite): "Capturing screenshot." (announced on entry.)

### State 3 — Capturing (long, ≥ 5 s)

- Same as State 2, except metadata row now renders `ElapsedCounter`: amber, JetBrains Mono, format `Still working… 6 s`. Updates every second.
- ARIA live (polite): "Still working. 6 seconds elapsed." (announced at 5 s, 10 s, 20 s, 30 s — not every second, to avoid SR spam.)

### State 4 — Ready (successful capture)

- Capture button: re-enabled, label reverts to "Capture screenshot" (allowing a new capture to replace the current).
- Preview: frame renders `<img src={`data:image/png;base64,${imageBase64}`}>` at full width, `object-contain`. Border solid.
- Metadata row (below preview): 3 stacked mono rows:
  - `file       acme-marketing_home_20260422-1014.png`
  - `size       1440 × 900`
  - `captured   2026-04-22 10:14`
- Secondary: Copy and Download **enabled** (opacity 100 %, amber on focus).
- ARIA live (polite): "Screenshot ready. 1440 by 900 pixels. Copy and download now available."

**Copy sub-state (transient, ≤ 2 s):**
- Copy button label morphs to `✓ Copied` (amber check glyph) for 1.6 s, then reverts.
- ARIA live (polite): "Image copied to clipboard."

**Copy denied sub-state:**
- Below the secondary action row, a single-line inline message appears: `Clipboard denied — use Download instead.` in `text-amber-300`. Persists until next action.
- ARIA live (assertive): "Clipboard permission denied. Use Download instead."

### State 5 — Error

- Capture button: re-enabled (so the flow is never stuck per AC-5.5).
- Preview: frame replaced with an error composition:
  - Top: `StatusPill` with `AlertTriangle` icon + short code (`AUTH`, `NOT_FOUND`, `UPSTREAM`, `NETWORK`, `UNKNOWN`).
  - Middle: the PRD-specified message for that code (AC-5.1 through AC-5.4).
  - Bottom: `[ ↻ Retry    R ]` button, full-width, amber accent border.
- Metadata row: shows error code mono `err   upstream_unavailable`.
- Secondary: Copy and Download **disabled**.
- ARIA live (assertive): the full error message + suggested next step.

### Critical flows

**Flow A — Golden path (PRD US-1 + US-2 or US-3):**

1. Panel mounts → State 1 (Idle).
2. User clicks Capture (or presses `C`) → State 2 (Capturing).
3. After 5 s still in flight → State 3 (Long capture).
4. Response arrives OK → State 4 (Ready). ARIA announces.
5. User presses `Y` → Copy sub-state. Toast + ARIA.
6. (or) User presses `D` → download triggers, filename per FR-09. No state change.

**Flow B — Error + retry (PRD US-5):**

1. Panel mounts → State 1.
2. Capture → State 2 → response is `404`.
3. Panel enters State 5 with `NOT_FOUND` code and message "This page has not been saved yet. Save it first, then try again."
4. User saves the page externally, then clicks Retry (or `R`) → State 2 again.
5. Success → State 4.

**Flow C — First-capture on brand-new page (PRD US-4):**

Same as Flow A, except the user sees the persistent hint throughout and understands why pending edits aren't reflected. No additional UI required.

### Responsiveness — 320 / 360 / 400 px

The layout is fluid; all widths scale. Explicit checks:

| Viewport | Behavior |
|---|---|
| **320 px (hard floor)** | Panel gutter 12 px → content width 296 px. PrimaryButton full-width 296 px; kbd glyph stays inside with min 8 px gap from label. Secondary actions collapse from 2-col grid to **stacked** at widths < 340 px (Copy on top, Download below, each full-width 36 px tall). Context metadata: label column 56 px, value column fills remaining. Preview frame height at 16:10 ratio = 185 px. |
| **360 px (typical)** | Gutter 12 px → 336 px content. Secondary actions in 2-col grid (half-width, 12 px gap). Preview height 210 px. |
| **400 px (ceiling)** | Gutter 16 px → 368 px content. Secondary actions 2-col with 16 px gap. Preview height 230 px. Label column in context rows widens to 64 px. |

Nothing is hidden at any width — feature parity across the range. Vertical scroll is acceptable and expected when the preview is populated.

### 4.4 Accessibility notes (per component)

- **PrimaryButton / SecondaryButton** — `<button>` elements with `type="button"`. Labels are full text (not icon-only). `aria-disabled` for disabled states. `aria-keyshortcuts` attribute lists the shortcut for SRs (e.g., `aria-keyshortcuts="c"`). Focus ring is always visible (no `focus:outline-none` without replacement).
- **PreviewFrame** — when ready, the `<img>` has `alt={`Screenshot of ${siteName}/${pageName}, ${width} by ${height} pixels`}`. When empty/capturing, the frame has `role="status"` and a visually hidden text description. When error, `role="alert"`.
- **LiveRegion** — a visually hidden `<div aria-live="polite" aria-atomic="true">` for non-critical announcements ("Screenshot ready"), and a separate `aria-live="assertive"` region for errors.
- **MetadataRow** — wrapped in `<dl>`; `<dt>` for labels, `<dd>` for values. Allows SRs to read "site: acme-marketing".
- **KbdGlyph** — rendered as `<kbd>`. Decorative glyphs have `aria-hidden="true"`; the shortcut is announced via `aria-keyshortcuts` on the parent button instead.
- **Focus order** — strictly top-to-bottom: Capture → Copy → Download → Retry (when present). `Tab` forward, `Shift+Tab` backward.
- **Color contrast** — verified intents:
  - Primary text `neutral-100` on `#0F1115` → ~17:1 (AAA).
  - Secondary `neutral-300` on `#0F1115` → ~11:1 (AAA).
  - Muted `neutral-500` on `#0F1115` → ~5.6:1 (AA for body, flagged: only used for labels ≥ 11 px; explicitly tested).
  - Amber-400 on `#0F1115` → ~9:1 (AAA).
  - Amber-300 error text on `#0F1115` → ~11:1 (AAA).
  - kbd glyph `neutral-300` on `white/[0.04]` composite over `#0F1115` → ~10:1 (AAA).

## 5. Responsiveness and accessibility

### Breakpoints

No media queries for layout re-flow beyond one internal threshold:

- **`< 340px`** — Secondary actions stack vertically (Copy above Download), each full-width 36 px tall.
- **`≥ 340px`** — Secondary actions side-by-side (2-col grid), each 32 px tall.

Panel gutter shifts from `px-3` (12 px) to `px-4` (16 px) at `≥ 400 px`. No horizontal scroll at any width. Preview frame is always `w-full` with fixed aspect ratio 16:10.

### WCAG 2.1 AA commitments

- **1.3.1 Info and relationships:** semantic `<dl>`, `<button>`, `<kbd>`, `<img alt>`, `role="alert"` for errors.
- **1.4.3 Contrast (minimum):** all body and label text meets 4.5:1; large text and UI elements meet 3:1. Actual ratios well above minimum (see § 4.4).
- **1.4.11 Non-text contrast:** button borders and focus rings meet 3:1 against adjacent surfaces.
- **2.1.1 Keyboard:** every action reachable and activatable via keyboard. No mouse-only affordances.
- **2.1.2 No keyboard trap:** panel is inside an iframe; focus can exit via Tab past the last control. No custom trap logic.
- **2.4.3 Focus order:** matches visual order, top-to-bottom.
- **2.4.7 Focus visible:** 2 px amber ring with 2 px offset against graphite background. Never suppressed.
- **3.3.1 Error identification:** errors are both visually signaled (amber pill + icon) and announced via `aria-live="assertive"`.
- **4.1.3 Status messages:** state changes announced via `aria-live="polite"` regions without taking focus.

### Focus pattern

- Single focus ring treatment: `outline-none ring-2 ring-amber-500/70 ring-offset-2 ring-offset-[#0F1115]`.
- Applied to: all buttons, the `<img>` in ready state (focusable with `tabindex="0"` so SR users can land on it; alt text is the description).

### ARIA live regions

Two regions mounted at the root of the panel:

```
<div aria-live="polite" aria-atomic="true" className="sr-only" data-region="status" />
<div aria-live="assertive" aria-atomic="true" className="sr-only" data-region="alert" />
```

- **`status` region** announces: "Capturing screenshot" (on entry to State 2), "Still working, N seconds elapsed" (at 5 s, 10 s, 20 s, 30 s thresholds only), "Screenshot ready. W by H pixels. Copy and download now available." (on entry to State 4), "Image copied to clipboard." (on copy success).
- **`alert` region** announces: every error message verbatim ("This page has not been saved yet. Save it first, then try again.", etc.), and the clipboard-denied message.

### Keyboard map

| Key | Action | State availability |
|---|---|---|
| `Tab` / `Shift+Tab` | Move focus forward/backward | all |
| `Enter` / `Space` | Activate focused button | all |
| `C` | Capture (equivalent to clicking Capture button) | idle, ready, error |
| `Y` | Copy image to clipboard | ready only |
| `D` | Download PNG | ready only |
| `R` | Retry | error only |
| `Esc` | Clear to idle (nice-to-have per PRD § 11) | ready |
| `Shift+?` | Show keyboard shortcuts reference (toggles a footer-expanded list) | all |

Shortcuts are registered on the panel root via a `keydown` handler that ignores events when focus is inside a text input (PageShot has none in v1, but the rule future-proofs). Modifier-less letter shortcuts are intentional — they're the Raycast convention and acceptable here because the panel has no text-entry surface.

Current spec replaces the preview image with a dimmed version of the previous image during State 2 to avoid a blank flash. Confirm this is acceptable under FR-11's "previous capture's image … is replaced" language. Designer's reading: yes — it's replaced on success; the dim during in-flight is a transitional rendering, not retention.
