## 1. Executive Summary

**PageShot** is a Sitecore Marketplace custom app that adds a one-button **"take a screenshot of the current page"** control to the SitecoreAI Pages editor. Editors click the button, see a clean chrome-free preview of the page they are working on, and either copy the image to clipboard or download it as a PNG.

The feature is implemented as a single **Page Builder Context Panel** extension. The image is produced by calling the SitecoreAI **Agent API** `GET /api/v1/pages/{pageId}/screenshot` through a Next.js route handler that holds the OAuth automation-client credentials server-side.

PageShot is deliberately the smallest end-to-end Marketplace app that is still genuinely useful. It serves a dual purpose:

1. **Marketer utility.** Eliminate the everyday friction of OS-screenshotting the editor chrome and cropping it manually, a workflow every content editor does multiple times per day for Slack/Teams review threads.
2. **Dogfood of the agentic framework.** Exercise scaffold → extension-route wiring → Marketplace SDK client → SitecoreAI API auth → real-portal test app in one clean pass, surfacing any remaining friction in the newly-updated skill set.

## 2. Problem Statement

When a marketer working in SitecoreAI Pages needs to share what they are working on with a colleague — in Slack, Teams, email, or a review thread — they currently:

1. Switch to the OS screenshot tool (Snipping Tool / Cmd-Shift-4 / etc.).
2. Try to crop out the Pages editor chrome (left sidebar, top bar, sometimes the canvas ruler).
3. Save or copy.
4. Paste into the conversation.

This workflow is slow, cognitively disruptive (context switch away from the editor), inconsistent in quality (different croppings, different resolutions, sometimes captures of the editor UI instead of the page), and does not represent the page as it will actually render.

There is no first-party "capture the page I am editing" affordance in Sitecore Pages today. The Agent API `/screenshot` endpoint returns a clean render but is not surfaced in the editor UI.

## 3. Goals and Success Metrics

### Product goals (outcomes)

- **G1 — Reduce time-to-shared-screenshot.** An editor who needs to share what they're working on gets from "I want to show this" to "image in clipboard or downloaded" without leaving the editor and without any cropping step.
- **G2 — Eliminate editor-chrome noise from shared artefacts.** Screenshots produced by PageShot contain only the rendered page — no sidebars, no toolbars, no context panels — so no downstream cleanup is required.
- **G3 — Graceful behavior under real-world API conditions.** The experience remains comprehensible and recoverable when the Agent API is slow, the page isn't saved, credentials are misconfigured, or the browser blocks clipboard access — the user always has a clear next step.

### Product success criteria (golden path)

- **M1.** An editor in Pages can produce a page screenshot and copy or download it in **≤ 10 seconds wall-clock on a healthy tenant**, measured from clicking the panel's capture button to the image being in the clipboard or the download dialog.
- **M2.** **Zero additional clicks** beyond panel-open → capture → (copy | download).
- **M3.** The screenshot **does not contain any Pages editor chrome** (no left sidebar, no top bar, no context panel itself).
- **M4.** All documented error cases (Q8 of discovery) produce a clear, actionable inline message rather than a silent failure or a raw HTTP error.

### Dogfood success criteria (framework signal)

- **D1.** The build runs end-to-end from `setup/scaffold.md` → `marketplace-sdk/lifecycle.md` → `marketplace-sdk/extension-routes.md` → `marketplace-sdk/client.md` → `apis/sitecoreai/auth.md` → `apis/sitecoreai/agent-api.md` with **zero mid-run manual edits to skill files**.
- **D2.** The first successful screenshot from a real deployed Vercel preview app happens without needing to debug the OAuth/client-credentials flow beyond the checklist in `auth.md § 0`.
- **D3.** Any friction encountered is captured as a **patch-queue entry** in `.agent/skills/sitecore/marketplace-sdk/CATALOG.md` and classified (skill gap / command gap / external bug).

## 5. Product Scope

### In Scope

1. Single **Page Builder Context Panel** extension in a custom Marketplace app.
2. Subscribe to `pages.context` via `@sitecore-marketplace-sdk/client` and obtain the current `pageId`, site name, and page name.
3. A single **"Capture screenshot"** button that:
   - Calls a Next.js API route (`/api/screenshot/[pageId]`).
   - The route proxies to Agent API `GET /api/v1/pages/{pageId}/screenshot` using server-side OAuth client credentials.
   - Returns the base64 image to the panel.
4. Inline preview of the returned image.
5. **Copy image to clipboard** action (`navigator.clipboard.write` with `image/png` blob).
6. **Download PNG** action, filename format:
   `{siteName}_{pageName}_{YYYYMMDD-HHmm}.png` — lower-kebab-case, ASCII only, max 100 chars (truncate site/page if needed, preserving extension).
7. Loading state with spinner and disabled button.
8. Elapsed-time counter that appears after 5 s to signal the request is still running.
9. Documented error handling (see § 7).
10. Custom-app registration in the target Sitecore tenant.
11. Deployed to Vercel; production-URL test app registered in Cloud Portal.

### Out of Scope

- Annotations, drawing, arrows, or text overlays on the captured image.
- Cropping or zoom / region selection.
- History of past screenshots.
- Batch capture of multiple pages.
- Choosing a specific viewport / breakpoint for capture (endpoint default only).
- Direct "share to Slack / Teams / email" integrations.
- Image editing beyond what the browser's clipboard API supports.
- Public-app submission to the Marketplace (custom app only in v1).
- Local HTTPS certificate setup — Vercel preview URLs are the integration-test surface. Local dev is HTTP for UI iteration only.

## 6. User Stories

### US-1 — Capture the current page

> **As a** content editor working on a page in SitecoreAI Pages
> **I want** to capture a clean screenshot of that page with one click
> **So that** I can paste it into Slack or Teams for a colleague to review without leaving the editor.

**Acceptance criteria:**
- `AC-1.1` When the context panel is open on an existing saved page, a **"Capture screenshot"** button is visible and enabled.
- `AC-1.2` Clicking the button disables it and shows a spinner until the response arrives.
- `AC-1.3` When the response arrives successfully, the image is rendered inline in the panel at a reasonable preview size (width ≤ panel width).
- `AC-1.4` The captured image contains only the rendered page, no Pages editor chrome.
- `AC-1.5` On a healthy tenant under normal load, end-to-end capture completes within 10 seconds of the click.

### US-2 — Copy to clipboard

> **As a** marketer who just captured a screenshot
> **I want** to copy the image to my clipboard with one click
> **So that** I can paste it directly into a chat message.

**Acceptance criteria:**
- `AC-2.1` After a successful capture, a **"Copy image"** button is visible and enabled.
- `AC-2.2` Clicking it writes the image to the clipboard as a PNG blob (not a data URL, not a file path).
- `AC-2.3` A brief "Copied" confirmation (toast or inline label) appears for ≤ 2 seconds.
- `AC-2.4` If the browser denies clipboard permission, an inline message explains what happened and suggests Download as fallback.

### US-3 — Download as PNG

> **As a** marketer who just captured a screenshot
> **I want** to download the image as a PNG
> **So that** I can attach it to an email or keep it for later.

**Acceptance criteria:**
- `AC-3.1` After a successful capture, a **"Download PNG"** button is visible and enabled.
- `AC-3.2` Clicking it triggers a browser download using filename `{siteName}_{pageName}_{YYYYMMDD-HHmm}.png`.
- `AC-3.3` Filename uses lower-kebab-case ASCII only; unsafe characters are replaced with `-`; length truncated to 100 chars preserving the `.png` extension.
- `AC-3.4` Timestamp in the filename is the capture time (local time on the user's machine).

### US-4 — Understand unsaved-draft state

> **As a** marketer with unsaved edits on the current page
> **I want** to understand that the screenshot shows the last saved version, not my draft
> **So that** I am not confused when my pending changes don't appear in the image.

**Acceptance criteria:**
- `AC-4.1` Whenever the panel is rendered for a page, a small static hint label reads: **"Screenshot shows the last saved version of this page."**
- `AC-4.2` The capture button remains enabled regardless of the editor's saved/dirty state.
- `AC-4.3` If the backend returns a "page not found" (`404`) — e.g. page has never been saved — the error message suggests **"Save the page first, then try again."**

### US-5 — Recover from transient failure

> **As a** marketer whose screenshot call fails (timeout, 5xx, network)
> **I want** an obvious, actionable error message and a clear retry path
> **So that** I can recover without opening dev tools or restarting the editor.

**Acceptance criteria:**
- `AC-5.1` `401` from the Agent API triggers one automatic token refresh + retry. If still `401`, the panel shows **"Authentication failed. Ask your administrator to check the app's credentials."**
- `AC-5.2` `404` shows **"This page has not been saved yet. Save it first, then try again."**
- `AC-5.3` `5xx` and timeout show **"Screenshot service is temporarily unavailable. Try again in a moment."** with a visible "Retry" button.
- `AC-5.4` Network offline is detected (`navigator.onLine === false` or fetch rejection) and shows **"You appear to be offline."**
- `AC-5.5` No error state leaves the panel in an unrecoverable state — a retry is always one click away.

## 7. Functional Requirements

| ID | Requirement |
|---|---|
| **FR-01** | The app exposes a single extension point: **Page Builder Context Panel**. |
| **FR-02** | The panel subscribes to `pages.context` via `@sitecore-marketplace-sdk/client` and uses the event's `pageId`, `siteName`, and `pageName`. |
| **FR-03** | The panel renders a capture button, loading states, the captured image preview, and the Copy / Download actions. |
| **FR-04** | The panel **does not** hold or request `client_id` / `client_secret` in-browser. Credentials exist only server-side. |
| **FR-05** | The app provides a Next.js route handler at `/api/screenshot/[pageId]` that: (a) obtains a cached OAuth JWT via the shared token-cache pattern in `apis/sitecoreai/auth.md § 5`, (b) calls Agent API `GET /api/v1/pages/{pageId}/screenshot`, (c) returns the base64 image as JSON `{ image: string }` or an error envelope. |
| **FR-06** | On `401` from the Agent API, the route invalidates the cached token and retries once before returning an error. |
| **FR-07** | On any non-200 from the Agent API, the route returns an envelope `{ error: { code, message } }` where `code` is one of `auth`, `not_found`, `upstream_unavailable`, `network`, `unknown`. |
| **FR-08** | Copy-to-clipboard uses `navigator.clipboard.write([new ClipboardItem({ "image/png": blob })])`. |
| **FR-09** | Download uses the filename template specified in § 5.6. |
| **FR-10** | While a capture is in flight, the capture button is disabled; after 5 s, an elapsed-seconds counter appears ("Still working… 6 s"). |
| **FR-11** | On successful capture, the previous capture's image and state are replaced. Previous images are not retained. |
| **FR-12** | The static hint "Screenshot shows the last saved version of this page." is always visible whenever the panel is mounted with a valid `pageId`. |
| **FR-13** | OAuth credentials are loaded from environment variables `SITECORE_DEPLOY_CLIENT_ID` and `SITECORE_DEPLOY_CLIENT_SECRET` server-side only. If either is missing at runtime, the `/api/screenshot/[pageId]` route returns an `auth` error, and the panel shows the same admin-configuration message as `AC-5.1`. |

## 8. Non-Functional Requirements

- **NFR-P-01 — Perceived performance.** Golden-path capture ≤ 10 seconds wall-clock on a healthy tenant.
- **NFR-P-02 — Responsiveness.** Panel UI renders within 300 ms of iframe load; click feedback (button disable + spinner) within 100 ms of button press.
- **NFR-S-01 — Secret handling.** `client_id` / `client_secret` never reach the browser or any HTML response body. Only the server-side Node runtime reads them.
- **NFR-S-02 — Token caching.** JWT cached in server memory per the pattern in `apis/sitecoreai/auth.md § 5`, with ≥ 60 s expiry safety margin.
- **NFR-S-03 — No persistent storage of images.** Captured images are transient — held only in the panel's React state and the user's clipboard / downloads.
- **NFR-A-01 — Accessibility.** All interactive elements keyboard-operable; visible focus ring; button labels and states announced via ARIA live region updates on state change.
- **NFR-Br-01 — Browsers.** Latest Chrome, Edge, Firefox, Safari (last two major versions each). Clipboard API falls back gracefully on browsers without `ClipboardItem` image support.
- **NFR-Co-01 — Compliance with Marketplace iframe constraints.** All responses and the Next.js route serve `Content-Security-Policy` and `X-Frame-Options` compatible with being loaded inside Sitecore Pages. No new cookies that would break in the Marketplace iframe.
- **NFR-O-01 — Observability.** Server route logs structured entries (no secrets) for: token fetch, Agent API call status, errors. Logs retained per Vercel defaults.
- **NFR-R-01 — Rate-limit behavior.** Caller backs off on `429` / `5xx` from the Agent API and surfaces the UX-5 error state; see `apis/sitecoreai/agent-api.md` pitfalls.

## 9. Technical Considerations

### Stack

- **Framework:** Next.js (App Router), Node runtime for the route handler.
- **Marketplace SDK:** `@sitecore-marketplace-sdk/client` for iframe + `pages.context` subscription.
- **API surface consumed:** SitecoreAI Agent API `GET /api/v1/pages/{pageId}/screenshot`.
- **Auth:** OAuth 2.0 client credentials per `.agent/skills/sitecore/apis/sitecoreai/auth.md` (token cached server-side).
- **Hosting:** Vercel. Production URL registered as the panel's extension URL in the Cloud Portal custom app.
- **TLS:** HTTPS via Vercel defaults. **No local HTTPS cert setup** — local dev runs on plain HTTP for UI iteration; all portal-integrated testing happens on Vercel preview URLs.

### App architecture

- **Client-side iframe app** per `marketplace-sdk/lifecycle.md`, extended with a **server route handler** in the same Next.js app to hold the secret. This is explicitly the "client-side + thin server proxy" pattern — not a full-stack app in the sense of server-rendered pages.

### Env vars

```
SITECORE_DEPLOY_CLIENT_ID=<from Cloud Portal automation client>
SITECORE_DEPLOY_CLIENT_SECRET=<from Cloud Portal automation client>
```

Both are **server-only**. Not prefixed with `NEXT_PUBLIC_`.

### SDK-first rule

Page context comes from `@sitecore-marketplace-sdk/client`'s `pages.context` subscription, **not** from the Agent API's site/page lookup endpoints. Only the screenshot capture falls back to direct Agent API, because no SDK wrapper covers it.

### Custom-app registration

- Cloud Portal → target tenant → **Custom apps** → **Create**.
- Extension point: **Page Builder Context Panel**.
- Panel URL: the Vercel production URL for the `/panel` route.
- Scopes: minimum required for the Agent API screenshot endpoint.

## 10. Data Model Overview

PageShot stores no persistent data. Transient in-memory shapes only:

- **Panel state (React):**
  ```ts
  type PanelState =
    | { kind: "idle" }
    | { kind: "capturing"; startedAt: number }
    | { kind: "ready"; imageBase64: string; siteName: string; pageName: string; capturedAt: Date }
    | { kind: "error"; code: "auth" | "not_found" | "upstream_unavailable" | "network" | "unknown"; message: string };
  ```

- **Server-route response envelope:**
  ```ts
  type ScreenshotResponse =
    | { ok: true; image: string } // base64 PNG, no data-URL prefix
    | { ok: false; error: { code: "auth" | "not_found" | "upstream_unavailable" | "network" | "unknown"; message: string } };
  ```

- **Server-side token cache (process memory):** `{ token: string; expiresAt: number }` per `auth.md § 5`.

No database, no persistent file storage, no telemetry pipeline in v1.

## 11. UX Considerations

### Layout (Page Builder Context Panel)

```
┌──────────────────────────────────┐
│ PageShot                         │
│                                  │
│ [  Capture screenshot  ]         │
│                                  │
│ ℹ Screenshot shows the last      │
│    saved version of this page.   │
│                                  │
│ ── preview area ───────────────  │
│ │                              │ │
│ │     (image or empty state)   │ │
│ │                              │ │
│ ────────────────────────────    │
│                                  │
│ [ Copy image ]  [ Download PNG ] │
└──────────────────────────────────┘
```

### States

- **Idle:** buttons disabled except Capture; preview area shows subtle empty state "Click Capture to take a screenshot of this page."
- **Capturing:** Capture button disabled + spinner; after 5 s, "Still working… 6 s, 7 s…" updates every second.
- **Ready:** preview shown; Copy + Download enabled; Capture re-enabled to allow another capture (replaces the current image on success).
- **Error:** preview area shows error icon + message + Retry button. Copy + Download disabled. Static hint still visible.

### Tone

Functional, not playful. This is a tool, not a feature to merchandise. Error messages tell the user what to do next, not what went wrong internally.

### Keyboard

- `Tab` through Capture → Copy → Download.
- `Enter` / `Space` activates the focused button.
- `Esc` on the preview area clears to idle (optional, nice-to-have).

## 13. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| **R-1** Agent API `/screenshot` is slow (> 10 s) on realistic pages | Medium | Breaks M1 | Surface elapsed-time counter (FR-10); document observed latency in ship report; no artificial timeout shorter than Vercel's default. |
| **R-2** Client-credentials flow setup fails (wrong scopes, wrong audience, role issue) | Medium | Blocks all screenshots | Follow `auth.md § 0` credential-acquisition protocol; fail loudly with admin-actionable message (FR-13). |
| **R-3** Clipboard write denied by browser permissions | Medium | Copy action breaks | Graceful inline message + always-available Download fallback (AC-2.4). |
| **R-4** Iframe / cookie issues in Pages editor after deploy | Low | Panel doesn't load | Use Marketplace SDK testing checklist in `marketplace-sdk/testing-debug.md`; test on Vercel preview before production registration. |
| **R-5** Unsaved-draft confusion | Medium | Editor thinks tool is broken | Static hint (FR-12) + 404 handler with actionable message (AC-4.3). |
| **R-6** Vercel env-var misconfiguration (wrong tenant secret in production env) | Low | Auth fails silently or succeeds against wrong tenant | Checklist item in runbook; log tenant identifier (not secret) on first successful auth. |
| **R-7** Dogfood-first-run reveals skill gaps | **High (expected)** | Patch candidates pile up | That **is** the run's purpose — capture them in catalog per D3. |

## 14. Open Questions

- **OQ-1** Does the Agent API `/screenshot` support a `viewport` or `breakpoint` query parameter? Deferred: v1 uses the endpoint default; revisit if the default produces the wrong viewport in practice.
- **OQ-2** Which Sitecore tenant / organization will host the custom-app registration? Resolved at scaffold time, not at PRD time.
- **OQ-3** Vercel project name → repo convention `pageshot`. Resolved.

**Known non-issues accepted for this build** (raised in critical review Round 1, accepted as-is for a dogfood context):
- "Healthy tenant" in M1 is intentionally unfalsifiable — PageShot is a dogfood build, not a validated product release; M1 is a design intent, not a measured SLA.
- Dogfood metrics (D1–D3) and product metrics (M1–M4) can contradict in edge cases. Trade-offs are resolved case-by-case during the run rather than by a pre-committed priority.
- Error handling, accessibility, keyboard navigation, and the elapsed-time counter are in v1 scope rather than deferred; the purpose of this build includes exercising the full skill chain for those surfaces, not minimizing scope.
