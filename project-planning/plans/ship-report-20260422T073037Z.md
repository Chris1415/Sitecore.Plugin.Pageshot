## 1. Summary

PageShot shipped its v1 goal: a Sitecore Marketplace custom app that puts a one-button "Capture screenshot" control inside the SitecoreAI Pages editor, so a content editor can produce a clean chrome-free render of the page they are working on and immediately act on it — without leaving the editor and without cropping anything out by hand. What was delivered is the full end-to-end panel — a Shutterbug-direction Blok-tokened UI with aperture-styled hero shutter, polaroid preview card, and a three-pill action bar (Copy, Download, Open) — driven by a `pages.context` subscription on the Marketplace SDK and a single Node-runtime Next.js route handler at `/api/screenshot/[pageId]` that holds the OAuth `client_id` / `client_secret` server-side, retries once on `401`, and maps every upstream failure to a discrete error code. Beyond PRD minimum, the build also added a theme switcher (auto / light / dark), a multi-viewport capture toggle, height presets, and client-side auto-trim of the bottom padding — all of which remain inside the single-extension, stateless, custom-app envelope ADRs 0002–0006 locked in. The suite closes with **151/151 automated tests green across 18 files** and an NFR-S-01 secret-containment grep clean against the client bundle.

## 4. Known Limitations

Things that work as designed but have real caveats a reader should know about.

**Big:**

- **(a) Download is blocked by the Sitecore Pages iframe sandbox.** The `<a download>` click fires correctly and the browser would honour it, but the parent iframe's `sandbox` attribute does not currently include `allow-downloads`. The same click outside the Sitecore iframe (e.g. on the raw Vercel URL) saves the file as expected. Workaround: the panel exposes an **Open in new tab** action (`components/use-open-image.ts` → `window.open(url, '_blank', 'noopener,noreferrer')`) — the editor right-clicks-save from the new tab. Resolution depends on Sitecore adding `allow-downloads` to the iframe sandbox; zero code change required on our side once they ship it.
- **(b) Agent API `height` parameter is exact, not a `fullPage: true` toggle.** The SitecoreAI Agent API accepts only a numeric `height` in pixels — there is no `fullPage` mode that measures the document and returns the full height. This produces a common failure mode: you request `height=4000`, the page is 2800 px tall, and the bottom 1200 px of the returned image is padding / blank canvas. Mitigation: the panel surfaces a **Height** preset picker (short / medium / tall) and, after the response, runs **client-side auto-trim** of the bottom padding (`lib/trim-image.ts`, 9 safety guards, returns original on any ambiguity). This solves the user's experience but it does not solve the underlying API gap. Logged as SitecoreAI API feedback item.
- **(c) Single-tenant custom app, not public Marketplace.** PageShot v1 is intentionally **not** submitted to the public Sitecore Marketplace (ADR-0005). Every tenant that wants PageShot must configure its own custom-app registration, its own OAuth automation client, and its own Vercel env vars. There is no "install from Marketplace" flow.

**Smaller:**

- Copy-to-clipboard requires an explicit browser permission grant on some browsers (e.g. Firefox may block programmatic image writes). The app detects the rejection, surfaces `CLIPBOARD_DENIED_MESSAGE` inline, and falls back to Download (or Open-in-new-tab). No silent failure.
- Playwright E2E smoke is installed but **not wired into CI**. Release-readiness was validated by the stakeholder interactively inside real Sitecore Pages + the automated unit/component/integration suite (151 tests). Adding a headless Playwright smoke to CI would shorten the regression cycle for future changes (see § 8 recommendation (2)).
- The token cache is **process-memory only** (ADR-0006). Each Vercel serverless cold-start pays one ~200 ms OAuth round-trip. If multi-instance scaling ever makes this painful, the remedy is a shared KV cache — which would require a new ADR superseding the in-memory token-cache choice.
- "Healthy tenant" in M1 is an unfalsifiable design-intent target, not a measured SLA — PRD § 14 "Known non-issues accepted for this build" already documents this.

Small, non-urgent.

## 8. Recommendations for Next Iteration

Concrete, prioritized, informed by what this run actually discovered:

1. **Retire the Open-in-new-tab button once Sitecore ships `allow-downloads` in the Pages iframe sandbox.** The action exists purely as a today-workaround for § 4 (a). When the sandbox fix ships, Download works natively and the third pill can come out. Watch the Sitecore release channel; one-line panel change.
2. **Wire Playwright E2E smoke into CI.** The Playwright scripts exist but run only on demand. Adding a headless `pnpm test:e2e` step that hits a mock server (returning a known base64 PNG) would catch regression in the capture → render → action wiring across future changes, and it is the last gap in the automated-coverage story (§ 5).
3. **Consider a version-selector in the panel.** The Agent API call currently hardcodes `version=1`; there is no UI affordance for selecting a newer API version once Sitecore ships one. A simple dropdown in a settings slot would avoid a future code change when `v2` lands — and would set precedent for how the panel surfaces Agent API evolution.
4. **File three SitecoreAI API feedback tickets (already logged in `.agent/skills/sitecore/marketplace-sdk/CATALOG.md` Run 10).** The three API gaps encountered during this run — (a) `version` parameter required as integer not string, (b) response field documented as `image` but actually returned as `screenshot_base64`, (c) no `fullPage: true` toggle, only exact `height` — are upstream issues that cost us real debugging time. Turning those catalog notes into actual Sitecore-side tickets is high-leverage for every future Marketplace app that hits the same endpoints.
5. **If public-Marketplace submission ever becomes a goal, supersede ADR-0005 with a new ADR and add review-readiness tasks.** Branding, support docs, per-tenant install UX, submission prep — none of this is in v1 and none of it should be retrofitted silently. A fresh PRD + ADR keeps the audit trail clean.
