## Critical findings (must fix)

### C1 — Lint error in `useThemeMode.ts` blocks lint gate
- **File:** `components/useThemeMode.ts` L70–73
- **Category:** correctness / quality (ESLint `react-hooks/set-state-in-effect`)
- **Severity:** critical
- **Description:** The mount-time re-read effect calls `setModeState` synchronously inside a `useEffect` body, which React 19 flags as a cascading-render anti-pattern. The effect is also fundamentally redundant: `useState(() => readStoredMode())` already runs `readStoredMode()` on the first render, so there's no second read to reconcile. `npm run lint` exits with code 1 because of this rule.
- **Fix:** Delete the redundant `useEffect` block entirely. The initial-state lazy initializer already provides the correct value.

## Major findings (should fix)

### M1 — `tsconfig.json` missing `noUncheckedIndexedAccess: true`
- **File:** `tsconfig.json` L3–10
- **Category:** architecture compliance / quality
- **Severity:** major
- **Description:** § 4c-3 specifies the TypeScript strict configuration must include `noUncheckedIndexedAccess: true`. This flag catches undefined-array-element accesses at compile time and is the contract the rest of the codebase was written against (see `lib/filename.ts` and others that already use `!` non-null assertions assuming the flag is on).
- **Fix:** Add `"noUncheckedIndexedAccess": true` to `compilerOptions`. Verify typecheck is still clean.

### M2 — OAuth 4xx mis-maps to `upstream_unavailable` instead of `auth`
- **File:** `app/api/screenshot/[pageId]/route.ts` L193 (and L215 for the retry branch)
- **Category:** correctness / security-adjacent
- **Severity:** major
- **Description:** When the SitecoreAI token endpoint returns a non-2xx (e.g. bad credentials, revoked client, wrong audience), `fetchFreshToken` throws `SitecoreTokenFetchError`. The route's first `catch` discriminates only on `SitecoreTokenConfigError` and lumps every other error into `upstream_unavailable`. An HTTP 400/401/403 at the auth endpoint is categorically an **auth-credentials** problem, not a transient upstream outage — the user-facing copy ("Try again in a moment.") is misleading and the admin has no indication that their client_id/secret is rejected. § 4c-6 and FR-13 require `{ code: 'auth' }` for credential rejection.
- **Fix:** Distinguish `SitecoreTokenFetchError` with a 4xx `status` → `errorEnvelope('auth', 500)`. Network-layer failures (status === 0) stay as `upstream_unavailable`. 5xx from auth endpoint stays as `upstream_unavailable`.

### M3 — `useDownloadImage` recomputes `download` callback on every status change
- **File:** `components/use-download-image.ts` L114
- **Category:** performance / quality
- **Severity:** major
- **Description:** `status` is in the `useCallback` deps — but `setStatus` is called inside the callback, so every status transition re-creates `download`, defeating memoization and propagating to any caller effects that depend on it. The read of `status` inside the callback is used only for the `'downloading'` re-entrancy guard; this is better served by a ref.
- **Fix:** Track in-flight via a ref (`downloadingRef`) and drop `status` from the deps.

### M4 — `useOpenImage` has the same status-in-deps issue
- **File:** `components/use-open-image.ts` L71
- **Category:** performance / quality
- **Severity:** major
- **Description:** Same bug pattern as M3. `status === 'opening'` guard plus `status` in deps.
- **Fix:** Same — use an in-flight ref, drop `status` from deps.

### M5 — `ShutterWithButtonRef` / `ActionPillWithButtonRef` run effect every render
- **File:** `components/PageshotPanel.tsx` L561-563 and L587-590
- **Category:** performance / quality
- **Severity:** major
- **Description:** Both wrapper components declare a `useEffect` with no dependency array, so the effect runs after every single render of these components (which re-render whenever any panel-level state changes). The effect's job is to copy a `<button>` ref to a forwarded `buttonRef.current`. Since the DOM button doesn't move, this is mostly wasted work — but it also means the ref write happens repeatedly in each render cycle, which could race with React's commit phase.
- **Fix:** Convert to a ref callback on the wrapping `<div>` that captures the inner button once — no `useEffect` needed. Alternatively, forward the ref directly to `<Shutter>` / `<ActionPill>` via a proper `React.forwardRef` or `ref` prop.

### M6 — `/` landing page ships scaffold demo content unrelated to PageShot
- **File:** `app/page.tsx` + `components/examples/**`
- **Category:** quality / dead code
- **Severity:** major
- **Description:** The root route renders an "Application Context" + "ListLanguagesFromClientSdk" scaffold demo, pulling in Collapsible + Card + Badge + XMC query machinery that PageShot itself does not need. In the custom-app registration, the panel surface is `/panel` — `/` is effectively dead. The scaffold examples also contain `console.log` debugging statements (L50, L53 of `list-languages.tsx`) — in production these log the editor's language list to the browser console. Low-risk but unnecessary noise and a bundle-size tax for every panel load (the layout eagerly imports MarketplaceProvider, so the full tree resolves).
- **Fix:** Replace `app/page.tsx` with a minimal placeholder that either redirects to `/panel` or renders a short copy directing users to the panel surface. Remove the `components/examples/` directory and its console.logs. This is scope-narrowing not feature-adding — the whole subtree is a scaffold leftover.

### M7 — `pre-QA → post-QA` trail: scaffold `console.log` leaks into production
- **File:** `components/examples/built-in-auth/with-xmc/list-languages.tsx` L50, L53
- **Category:** security / quality (NFR-O-01 adjacent)
- **Severity:** major
- **Description:** Two bare `console.log(languagesResponse)` / `console.log("error from client", err)` statements live in the scaffold-example code. If this file is retained in production (see M6), those logs run on every page load where the example is mounted. § 4c-1 explicitly forbids logging secrets, and while the language list is not a secret, production code should not ship demo `console.log`s. If M6 is fixed by deleting `examples/`, this is subsumed.
- **Fix:** Subsumed by M6's directory removal.

### M8 — Catch-all `catch {}` in `issueCapture` mis-labels programming errors as network errors
- **File:** `components/PageshotPanel.tsx` L256-262
- **Category:** correctness / quality
- **Severity:** major
- **Description:** The outer `try` wraps `Promise.all(selected.map(...))` plus `trimBottomPadding` plus `dispatch`. Any throw inside any of those — a TypeError from a malformed envelope, an unexpected exception from `trimBottomPadding`, a React devtools invariant — is mapped to `{ code: 'network', message: "Check your connection, then try again." }`. This actively hides bugs from the editor and from any future observability pipeline.
- **Fix:** Narrow the try to just the fetch, and assert programming-error paths via unreachable sentinels that log to `console.error` (not silently convert to network). The sentinel `throw new Error('unreachable')` on L242 is a concrete example of an error that would be silently classified as "you're offline" — misleading.

### Mn4 — `Shutter.tsx` uses `h-36 w-36` wrapper (144px) for a 112px button
- **File:** `components/Shutter.tsx` L100
- **Category:** quality
- **Severity:** minor
- **Description:** The outer wrapper is `h-36 w-36` (36 × 4 = 144 px) while the button itself is `h-28 w-28` (112 px). The extra 32 px is intentional for the bloom overlay (absolutely positioned `inset-0`), but the spec § 4c-4 describes a 112 × 112 hero. Not wrong — the bloom effect needs headroom — but could be documented inline or rewritten to avoid confusion with the spec.

### Mn5 — `decodeTenantIdFromJwt` does not fail-closed on oversized payloads
- **File:** `lib/sitecore-token.ts` L140-160
- **Category:** security-adjacent
- **Severity:** minor
- **Description:** The JWT payload decode runs `JSON.parse` on whatever the token endpoint returned. A maliciously crafted token could return a 10 MB base64url payload and force the Node process to parse it. In practice the attacker would have to compromise `auth.sitecorecloud.io`, which is far outside PageShot's threat model, but defence-in-depth would cap `payloadSegment.length` before decoding.

- **M8 — Narrow `issueCapture` `catch` to the fetch block only.**
  Previously a bare `catch {}` mapped every downstream exception (including invariants and programming errors from `trimBottomPadding` or `dispatch`) to `{ code: 'network' }`. The `unreachable` sentinel error inside the `trimmedBodies.map` was also upgraded from a bare `Error('unreachable')` to a descriptive `pageshot: expected ok:true envelope...` message.
  Commit: `e7a7c75 review-fix: narrow issueCapture catch to fetch failures only (M8)`
